# QuantAI: Security Audit Report (Penetration Tester Perspective)

## Executive Summary
This audit reviews the security posture of the QuantAI Trading System, specifically analyzing API key management, database connectivity, IAM permissions, infrastructure configuration (Terraform), and potential code-level vulnerabilities. Overall, the system exhibits a highly mature, production-grade security architecture for a solo-developer project, heavily leveraging GCP Secret Manager, Workload Identity Federation, and robust pre-commit secret scanning (`gitleaks`). However, several moderate-to-high severity risks remain regarding Docker supply chains and execution environment sandboxing.

---

## 1. API Key Handling Paths
**Status: EXCELLENT**
- Alpaca API keys (`ALPACA_API_KEY`, `ALPACA_SECRET_KEY`) and Telegram bot tokens are never hardcoded. 
- In the live Cloud Run path, they are injected securely via Google Secret Manager natively linked in Terraform (`value_source { secret_key_ref }`). The Python code reads them directly from the environment variables populated safely by GCP at runtime.
- **Risk:** If a developer runs the system locally via WSL, they must place these keys in a `.env` file. If `.gitignore` is bypassed, these could leak.
- **Mitigation in place:** The project enforces a strict `gitleaks` pre-commit hook that actively blocks commits containing API key patterns.

## 2. Database Connection Security
**Status: GOOD (with minor risks)**
- The Cloud SQL PostgreSQL instance enforces `ssl_mode = "ENCRYPTED_ONLY"`.
- Cloud Run Jobs connect via the Cloud SQL Auth Proxy over a Unix socket (`/cloudsql/PROJECT:REGION:INSTANCE`). This means no passwords or connections traverse the public internet; it remains entirely within Google's internal network.
- **Risk:** The raw `DATABASE_URL` secret (containing the cleartext random password generated by Terraform) is exposed to the Cloud Run Job's environment variables. If a malicious dependency executes `os.environ` and exfiltrates it, the DB is compromised.
- **Recommendation:** Implement PostgreSQL IAM Database Authentication to remove passwords entirely, relying strictly on the Service Account token to log in.

## 3. Secret Rotation Procedures
**Status: MODERATE**
- Passwords are generated via Terraform's `random_password` resource. Rotating the database password requires tainting the Terraform resource, running `terraform apply`, and restarting the Cloud Run Jobs.
- API keys are manually added as new versions to Secret Manager using the `gcloud secrets versions add` command. Since Cloud Run pulls `version = "latest"`, the rotation is effective upon the next daily run. 
- There is no automated rotation schedule.

## 4. Network Security
**Status: EXCELLENT**
- The Cloud SQL instance (`quantai-postgres`) has an empty `authorized_networks` list. Direct TCP connections (Port 5432) from the public internet are firewalled off.
- The Cloud Run Jobs do not expose an ingress (no HTTP triggers). They are exclusively executed via Cloud Scheduler's internal VPC POST request.
- The entire attack surface is effectively zero from the outside internet.

## 5. IAM Permissions Audit (Principle of Least Privilege)
**Status: VERY GOOD**
- A dedicated Service Account (`quantai-trading`) is utilized for all operations.
- Roles granted: `roles/run.jobsExecutorWithOverrides`, `roles/secretmanager.secretAccessor`, `roles/cloudsql.client`. 
- **Risk:** The Service Account is granted `roles/cloudsql.admin`. This is an excessively powerful role just to allow the script to start/stop the DB instance for cost optimization. If the SA is compromised, the attacker can delete the database entirely.
- **Recommendation:** Create a custom IAM role with only `cloudsql.instances.start` and `cloudsql.instances.stop` permissions, stripping the `admin` rights.

## 6. Injection Vulnerabilities
**Status: EXCELLENT**
- **SQL Injection:** All PostgreSQL queries in `alpaca_direct.py` and the `scripts/` directory use parameterized queries (e.g., `cur.execute("INSERT ... VALUES (%s, %s)", (val1, val2))`). There is no string concatenation `f"SELECT * FROM table WHERE id={user_input}"` found in the DB layer.
- **Command Injection:** Bash scripts (`run_daily.sh`, `backup_postgres.sh`) use static, hardcoded commands. They do not ingest unsanitized user inputs.

## 7. Dependency Vulnerabilities
**Status: HIGH RISK**
- Python relies on `requirements.txt` / `pip`. There is no evidence of a `Pipfile.lock` or `poetry.lock` enforcing strict hash-checking for dependency resolution.
- **Supply Chain Risk:** If `pandas`, `requests`, or `psycopg2` are hijacked on PyPI, the subsequent `docker build` will blindly pull the malicious version. 
- **Recommendation:** Implement `pip-compile` to generate a strict, hash-pinned `requirements.txt`. Enable GitHub Dependabot for automated CVE patching.

## 8. Logging Security
**Status: MODERATE**
- `logger.info()` and `logger.debug()` are used extensively. 
- **Risk:** In `alpaca_direct.py`, if an order fails, the raw `requests.HTTPError` body is logged (`body[:200]`). If Alpaca alters their error payloads to echo back the `APCA-API-KEY-ID` header in the error message, the API key will be permanently written to GCP Cloud Logging.
- **Recommendation:** Implement a custom LogFormatter to actively scrub regex patterns matching known secret formats before flushing to stdout.

## 9. Docker Image Security
**Status: MODERATE**
- Images are built via GitHub Actions and pushed to Artifact Registry.
- **Risk:** The `Dockerfile.runner` likely runs as `root` inside the container unless explicitly specified (`USER quantai`). While Cloud Run provides strong gVisor sandboxing, running as root violates defense-in-depth principles.
- **Recommendation:** Add a non-root user creation step in the Dockerfile and execute the Python scripts under that user.

## 10. Terraform State Security
**Status: CRITICAL FINDING**
- According to `gcp/terraform/main.tf`, the GCS backend configuration is commented out (`# backend "gcs" ...`). The state file is currently local (`terraform.tfstate`).
- **Risk:** The local state file contains the cleartext `random_password.cloud_sql.result`. If a developer's laptop is compromised, the DB password is immediately exposed. If multiple developers collaborate, local state causes desync and severe security risks.
- **Recommendation:** Immediately uncomment the GCS backend, enable Bucket Object Versioning, and apply strict IAM constraints on the bucket so only authorized administrators can read the raw state file.

## Recommendations Ranked by Severity

| Rank | Severity | Finding | Remediation Action |
| :--- | :--- | :--- | :--- |
| 1 | **CRITICAL** | Terraform local state exposes cleartext DB passwords. | Migrate Terraform state to a secured GCS backend immediately. |
| 2 | **HIGH** | Dependencies lack hash-pinning, exposing CI to supply chain attacks. | Use `pip-compile --generate-hashes` and enforce hash verification in the Dockerfile. |
| 3 | **HIGH** | Service Account possesses overly permissive `roles/cloudsql.admin`. | Create a custom IAM role restricting permissions strictly to DB start/stop. |
| 4 | **MEDIUM** | Cleartext `DATABASE_URL` exposed in container env vars. | Migrate to PostgreSQL IAM Authentication to remove passwords completely. |
| 5 | **LOW** | Container runs as root. | Implement `USER` directive in `Dockerfile.runner`. |
| 6 | **LOW** | Logs may inadvertently capture echoed secrets in API errors. | Sanitize HTTP error bodies before passing them to the logger. |
